# Internationalisation Guide — Tatomir

## Domain Separation

Three isolated i18n domains. Never mix strings between them.

```
packages/i18n/
  admin/      Admin interface strings      (apps/admin)
  editor/     Visual editor strings        (apps/studio)
  site/       Type helpers only            (site content lives in DB)
```

---

## Admin Namespaces

Path: `packages/i18n/admin/{locale}/{namespace}.json`

| File | Contents |
|------|----------|
| `auth.json` | Login, register, forgot password, 2FA, session expiry |
| `common.json` | Save, Cancel, Delete, Edit, Confirm, Back, Loading, Error, Empty, Success |
| `collections.json` | Collection builder, field types, field settings, entry states |
| `post-types.json` | Post type builder, taxonomy manager, post states, built-in post type labels |
| `media.json` | Upload widget, file type errors, folder management, bulk actions |
| `pages.json` | Pages list, page settings modal, slug editor, access control labels |
| `settings.json` | Site settings, domains, integrations |
| `users.json` | Team members, role labels (Owner, Admin, Editor, Viewer), invite flow |
| `members.json` | Site members, member groups, access rules, auth provider labels |
| `billing.json` | Plan names, feature comparison, upgrade prompts, invoice history |
| `notifications.json` | Toast messages, error states, success confirmations |
| `dashboard.json` | Widget labels, stats titles, quick action labels |
| `design-system.json` | Token editor labels, color/typography/spacing control labels |

**Rule:** A string belongs in a namespace when it appears exclusively in that feature. Strings appearing in 3+ features go in `common.json`.

| File | Contents |
|------|----------|

## Adding a New Language

1. Copy `packages/i18n/admin/en/` → `packages/i18n/admin/{locale}/`
2. Copy `packages/i18n/editor/en/` → `packages/i18n/editor/{locale}/`
3. Translate all values. Keys are never translated.
4. Add the locale code to `packages/i18n/config.ts` → `supportedLocales`.
5. Run `pnpm typecheck` — missing keys surface as TypeScript errors.
6. CI checks that all keys from `en` exist in the new locale.

Locale codes follow BCP 47: `en`, `fr`, `es`, `de`, `pt-BR`, `ja`, `zh-CN`.

## Site Content Translations

Stored in the database. Not static files.

- `Page.locale` — one row per locale per page.
- `CollectionEntry.locale` — one row per locale per entry.
- `PostEntry.locale` — one row per locale per post.
- `CmsVariable.values` — JSON object: `{ "en": "Contact us", "fr": "Contactez-nous" }`.
- `MediaFile.alt` / `caption` — per file, per locale (separate `MediaFileLocale` join if needed).

Missing locale falls back to the site's `defaultLocale`.

The renderer injects `hreflang` tags automatically for all available locales of a page.

---

## Site Auth Pages i18n

Auth pages (login, register, reset password, member dashboard) are regular Tatomir pages.

- Designed visually in the editor — one locale variant per supported locale.
- Auth form labels and error messages are stored as CMS variables per site.
- Translating auth strings = translating the CMS variables in the admin (no code, no files).

## Key Naming Conventions

```json
{
  "featureArea.componentName.string": "Value",
  "fieldTypes.richText": "Rich Text",
  "actions.save": "Save",
  "states.loading": "Loading...",
  "errors.required": "This field is required"
}
```

Rules:
- camelCase keys only
- Maximum 3 levels of nesting
- No abbreviations in keys
- Interpolation: `"welcome": "Welcome, {name}!"` — ICU message format (next-intl)
- Pluralisation: `"items": "{count, plural, one {# item} other {# items}}"`
